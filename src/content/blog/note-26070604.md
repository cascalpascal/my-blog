## 디자인패턴

### 생성 패턴 - FABPS

| 패턴 | 기능 | 특징 |
| --- | ---- | --- |
| Factory Method | 상위 클래스: 인터페이스 정의 <br> 서브 클래스: 실제 생성 | 확장성 |
| Abstract Factory | 서로 관련된 객체들의 **조합**을 만드는 API 제공 | Kit |
| Builder | 복잡한 객체를 생성하는 과정을 단계별로 분리 | 텔레스코핑 패턴 문제 해결 |
| Prototype | 원본 객체의 인스턴스를 복제해 새 객체를 생성 | 필요한 부분만 수정 |
| Singleton | 한 클래스에 단 하나의 객체만 존재하도록 강제 | 어디서든 참조 가능 |


<br>

---

**Factory Method** vs **Abstract Factory**  

단일 객체를 생성하는 것 vs 여러 제품군을 생성하는 것  
Creator 클래스가 상속을 통해 제품을 생성하는 방법을 변경 vs 구체적인 클래스를 알 필요 없이, 팩토리 교체로 전체 제품군을 변경

---

### 구조 패턴 - ABCDFFP
| 패턴 | 기능 | 특징 |
| --- | ---- | --- |
| Adapter | 서로 다른 인터페이스를 가진 클래스들을 연결 | Wrapper |
| Bridge | 서로 영향을 받지 않고 독립적으로 확장할 수 있게 추상부와 구현부를 분리 | Composition | 
| Composite | 객체를 트리 구조로 구성해 '부분-전체' 계층 표현 | 개별 객체와 복합 객체를 동일하게 취급 |
| Decorator | 인터페이스를 바꾸지 않고도 새로운 기능(책임) 추가 | 객체를 Wrap해서 장식 |
| Facade | 복잡한 서브시스템에 대한 간단한 통합 인터페이스 제공 | 클라이언트와 서브시스템 간의 결합도를 낮춤 |
| Flyweight | 객체 공유와 가상 인스턴스 제공으로 메모리 절약 | 클래스 경량화 |
| Proxy | 어떤 객체에 대한 접근을 제어하거나 추가적인 기능을 부여하기 위해 해당 객체의 대리 객체를 사용 | 대리자 |

---

**Adapter** vs **Decorator**  
인터페이스를 변경해서 클라이언트에서 필요로 하는 인터페이스로 적응시키기 위한 용도 vs 인터페이스를 바꾸지 않고 책임을 추가  
인터페이스 호환성 vs 기존의 객체를 장식

### 행위 패턴 - CSIM VOT(CCSSIIMM VOT)

| 패턴 | 기능 | 비유 |
| --- | ---- | --- |
| Chain of Responsibility | 요청을 여러 객체가 순서대로 처리하도록 연결 | 결합도를 낮춤 | 고객센터 |
| Command | 요청 자체를 객체로 캡슐화해 실행, Undo/Redo 기능 | 리모컨 |
| Strategy | 알고리즘을 객체로 캡슐화해 동적으로 교체 | |
| State | 객체의 상태에 따라 다른 동작 수행 | 신호등 |
| Iterator | 컬렉션의 내부 구조를 노출하지 않고 요소에 순차적으로 접근 | 유튜브 다음 영상 자동 재생 |
| Interpreter | 언어의 문법을 정의하고 해석 | 번역기 |
| Mediator | 객체 간 통신을 중재자가 관리 | 항공 관제탑 |
| Memento | 객체의 특정 시점 상태를 저장하고 복원 | Save Game |
| Visitor | 객체의 내부 상태에 접근할 수 있는 방법을 제공해 호스트 객체에 새로운 연산을 추가 | 세무사 |
| Observer | 한 객체의 상태가 변화하면 객체에 상속되어 있는 다른 객체들에게 변화된 상태를 전달 | 유튜브 구독 채널 알림 |
| Template Method | 기능의 뼈대는 상위 클래스에서 정의하고, 세부처리 구체화는 하위 클래스에서 재정의 | 라면 끓이기 |
